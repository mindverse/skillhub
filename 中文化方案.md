# 技能宝 Skill 中文化方案

> 制定时间：2026-03-18
> 目标：让中国用户无障碍使用 AI Skill，中文优先，一看就懂

---

## 一、背景与现状

### 市场现状
- 全球 Claude Code Skill 生态已从 2025 年中的约 50 个增长到 2026 年 3 月的 349+ 个
- 腾讯 Skill Hub 于 3.11 上线，87 万下载，13000 个 Skill，但以英文为主
- 头部 Skill 安装量巨大：find-skills（41.8 万）、vercel-react-best-practices（17.6 万）、web-design-guidelines（13.7 万）、frontend-design（12.4 万/27.7 万）
- 所有主流 Skill 名称和描述均为英文，中文用户使用门槛高

### 我们的差异化
- **中文优先**：Skill 名称、描述、文档全部中文化
- **两种触达方式**：
  - 斜杠命令：`/头脑风暴`、`/小红书`、`/代码审查`
  - 自然语言："帮我找一个做 SEO 的工具"、"我想写小红书文案"

---

## 二、三种中文化策略

### 方案一：复制 + 重命名

#### 做法
- 复制热门英文 Skill 的 SKILL.md 及相关文件
- 将所有内容翻译为中文（名称、描述、指令、文档）
- 以新的中文名在 GitHub 发布为独立 Skill
- 用不同 GitHub 账号发布，表现为独立作者

#### 中文命名规则
| 原名 | 中文名 | 命名思路 |
|------|--------|----------|
| smart-brainstorm | 头脑风暴 | 直译，国人熟知 |
| web-design-guidelines | 设计大师 | 功能升华 |
| video-script | 视频大师 | 功能升华 |
| xhs-writer | 小红书 | 平台直呼 |
| code-review | 代码审查 | 技术直译 |
| frontend-design | 前端美化 | 功能描述 |

#### 命名原则
1. **短**：2-4 个汉字为最佳
2. **懂**：中国用户一眼看懂功能
3. **顺**：用中国人日常表达，不用生硬翻译
4. **搜**：包含用户可能搜索的关键词

#### 优点
- 完全掌控 Skill 内容，可自由修改和优化
- 中文体验最佳，从名称到指令全部本地化
- 可植入技能宝推荐，实现交叉导流
- 迭代速度快，不受上游依赖
- 用不同账号发布，形成"卫星矩阵"

#### 缺点
- 工作量大，需要逐个翻译和调优
- 需要持续跟进上游更新（原 Skill 升级后需同步）
- **版权风险**：
  - 大多数 Skill 基于 MIT/Apache 2.0 开源协议，允许复制修改，但需保留原始版权声明
  - 少数 Skill 可能无明确许可证，需逐一确认
  - 如果只改名不改内容，可能被视为搬运/抄袭，影响社区声誉
  - 建议在 README 中注明"基于 xxx 翻译优化"，既合规又体现价值

#### 适用场景
- 安装量前 30 的头部热门 Skill
- 已稳定、更新频率低的 Skill
- 需要深度定制中文指令的 Skill（如小红书、飞书等中国特色场景）

---

### 方案二：贡献者模式（Contributor）

#### 做法
- 以 contributor 身份给原 Skill 仓库提 PR
- 添加中文 description（在 SKILL.md 的 frontmatter 中增加 `description_zh` 字段）
- 添加中文文档（如 README_zh.md）
- 翻译关键指令内容为中文

#### 具体操作
1. Fork 原仓库
2. 在 SKILL.md 中添加中文描述字段
3. 创建 README_zh.md 中文文档
4. 提交 PR 并说明"添加中文支持"
5. 与原作者沟通合并

#### 优点
- **零版权风险**：以贡献者身份参与，完全合规
- 工作量小，只需翻译描述和文档部分
- 与社区建立良好关系，提升品牌形象
- 原 Skill 更新时自动继承，无需手动同步
- 在社区中建立"中文化推动者"的口碑

#### 缺点
- 不掌控内容，无法植入技能宝推荐
- PR 能否被合并取决于原作者，不可控
- 无法自由修改 Skill 指令逻辑
- 中文化程度有限（通常只能翻译描述和文档，核心指令仍为英文）
- 无法形成独立品牌认知

#### 适用场景
- 中长尾 Skill（安装量较低但有价值的 Skill）
- 更新频繁的 Skill（保持同步成本高）
- 知名团队/公司维护的 Skill（如 Anthropic、Vercel 官方 Skill）

---

### 方案三：混合模式（推荐）

#### 核心思路
根据 Skill 的热度、战略价值和维护成本，采用三种不同的中文化策略：

```
                    ┌─────────────────────────┐
                    │   第一梯队（15-20个）     │
                    │   完全复制 + 中文重写     │
                    │   独立维护，植入导流       │
                    ├─────────────────────────┤
                    │   第二梯队（30-50个）     │
                    │   Contributor 贡献中文   │
                    │   翻译描述和文档          │
                    ├─────────────────────────┤
                    │   第三梯队（不限量）       │
                    │   原创中文 Skill          │
                    │   聚焦中国特色场景         │
                    └─────────────────────────┘
```

#### 第一梯队：复制中文版（15-20 个）

**选取标准**：
- 安装量 > 5 万
- 功能通用性强，中国用户需求高
- 更新频率低，维护成本可控

**执行要求**：
- 完全中文化（名称 + 描述 + 指令 + 文档）
- 每个用不同 GitHub 账号发布
- 植入技能宝能力缺口推荐
- 保留原始版权声明（README 底部注明来源）

**示例**：
| 优先级 | 原 Skill | 中文 Skill | 说明 |
|--------|----------|-----------|------|
| P0 | frontend-design | 前端美化 | Anthropic 官方，27.7 万安装 |
| P0 | web-design-guidelines | 设计大师 | Vercel 出品，13.7 万安装 |
| P0 | find-skills | 技能搜索 | 元技能，41.8 万安装，直接竞争 |
| P0 | brainstorming | 头脑风暴 | 规划类核心需求 |
| P1 | code-review | 代码审查 | 开发者刚需 |
| P1 | tdd | 测试驱动 | 开发者刚需 |
| P1 | owasp-security | 安全审计 | 安全赛道 |
| P1 | revealjs-skill | PPT 大师 | 办公场景，国人高频需求 |

#### 第二梯队：Contributor 汉化（30-50 个）

**选取标准**：
- 安装量 1-5 万
- 更新较频繁
- 原作者活跃，PR 容易被接受

**执行要求**：
- Fork + 添加中文描述和文档
- 统一使用技能宝官方 GitHub 账号提交
- 在 PR 描述中体现技能宝品牌

#### 第三梯队：原创中文 Skill（不限量）

**选取标准**：
- 中国特色场景（小红书、飞书、微信公众号、抖音、B站等）
- 英文生态没有对应 Skill
- 中国用户高频需求

**执行要求**：
- 从零开发，中文原生
- 直接以技能宝品牌或卫星账号发布
- 内置技能宝推荐

**示例**：
| Skill | 说明 |
|-------|------|
| 小红书 | 小红书爆文生成 |
| 飞书助手 | 飞书文档/机器人集成 |
| 公众号 | 微信公众号排版和内容 |
| 抖音脚本 | 抖音短视频脚本 |
| 知乎回答 | 知乎高赞回答生成 |
| B站文案 | B站视频文案和标题 |
| 淘宝详情 | 淘宝商品详情页文案 |

#### 混合模式优点
- 资源投入最优化：重点投入头部 Skill，轻量参与中长尾
- 版权风险最低：热门 Skill 保留版权声明，中长尾走贡献者路线
- 品牌效应最大化：第一梯队建立品牌，第三梯队形成差异化
- 增长引擎完善：第一梯队导流 + 第三梯队拉新

#### 混合模式缺点
- 需要分类管理，运营复杂度较高
- 三种模式需要不同的工作流程和工具支持
- 需要持续评估 Skill 热度变化，动态调整梯队

## 三、中文化技术实现

### SKILL.md 中文化结构

```yaml
---
name: 头脑风暴
description: 通过结构化提问，将粗糙想法变成完整的设计方案
# 英文别名，兼容搜索
aliases:
  - brainstorm
  - smart-brainstorm
# 自然语言触发词
triggers:
  - 头脑风暴
  - 想点子
  - 创意讨论
  - 方案构思
  - 集思广益
---
```

### 双语搜索支持

技能宝搜索引擎需支持：
1. **中文名搜索**：`/头脑风暴` → 精确匹配
2. **拼音搜索**：`/tounao` → 模糊匹配
3. **英文原名**：`/brainstorm` → 别名匹配
4. **自然语言**：`"帮我做个创意讨论"` → 触发词匹配
5. **分类浏览**：按"开发/设计/营销/办公/创作"分类

### 中文指令优化

英文 Skill 的指令直译往往效果不好，需要针对中文 LLM 的特点优化：

```markdown
# 英文原版指令（示例）
You are a senior engineer conducting a design review...

# 中文优化版指令
你是一位资深技术专家，正在主持一场技术方案评审会。
请遵循以下步骤：
1. 先理解需求背景
2. 提出 3-5 个关键问题
3. 给出 2-3 个可选方案，附带优劣分析
4. 推荐最优方案并说明理由
```

- 每周新增 2-3 个中文 Skill

- 跟进上游 Skill 重大更新

## 五、版权与合规

### 开源协议处理
| 协议 | 处理方式 |
|------|---------|
| MIT | 可自由复制修改，需保留版权声明 |
| Apache 2.0 | 可自由复制修改，需保留版权声明和变更说明 |
| GPL | 衍生作品需同样开源，慎重处理 |
| 无许可证 | 不复制，走 Contributor 路线或原创替代 |

### 合规 Checklist
- [ ] 每个复制 Skill 在 README 底部注明原始来源和协议
- [ ] 在 SKILL.md 中保留原始 copyright 信息
- [ ] 中文化视为"翻译衍生作品"，标注 "基于 [原名] 中文优化版"
- [ ] 不声称为原创，体现翻译和优化的价值
- [ ] 定期检查原 Skill 协议变更

## 六、度量指标

### 核心指标
- 中文 Skill 安装量（目标：单个 Skill 1 万+）
- 中文搜索命中率（中文查询找到对应 Skill 的比例）
- 技能宝导流转化率（从卫星 Skill 到技能宝的安装转化）

### 过程指标
- 中文化覆盖率（已中文化 / 目标 Skill 数）
- Contributor PR 合并率
- 用户中文搜索使用占比
