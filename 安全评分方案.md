# 技能宝安全评分机制方案

> 核心理念：让安全性"可感知"，类似纯净水"37重过滤"的营销思路。
> 用户看到安全评分，会天然产生信任感，这是技能宝与竞品的核心差异化。

---

## 方案一：基础评分（低成本）

### 评分维度

| 维度 | 权重 | 检查内容 | 满分 |
|------|------|----------|------|
| **代码安全性** | 35% | 静态代码分析，检查危险模式 | 35 |
| **权限合规性** | 25% | 是否请求过高权限 | 25 |
| **来源可信度** | 20% | 作者信誉、仓库质量 | 20 |
| **元数据完整性** | 10% | 描述、文档、许可证是否齐全 | 10 |
| **社区认可度** | 10% | Star数、Fork数、下载量 | 10 |

### 代码安全性检查清单（35分）

**高危模式（命中任意一项扣10分，最多扣35分）：**
- `rm -rf /`、`rm -rf ~`、`rm -rf *` 等破坏性删除命令
- `curl | bash`、`wget | sh` 等远程代码执行
- `eval()`、`exec()` 动态代码执行
- `os.system()`、`subprocess.call()` 不安全的系统调用
- 访问可疑域名（非主流域名的外部请求）
- Base64 编码的可疑代码块
- 硬编码的 API Key、密码、Token

**中危模式（命中任意一项扣5分）：**
- `sudo`、`chmod 777` 等提权操作
- 修改系统文件（`/etc/`、`/usr/`）
- 访问用户敏感目录（`.ssh/`、`.aws/`、`.env`）
- 网络请求到非 HTTPS 地址
- 写入全局配置文件

**低危模式（命中任意一项扣2分）：**
- 使用过时或已知不安全的依赖
- 缺少错误处理
- 日志中输出敏感信息

### 权限合规性（25分）

| 检查项 | 分值 |
|--------|------|
| 不需要 root/sudo 权限 | +10 |
| 不修改系统级配置 | +5 |
| 不请求网络权限（或仅请求必要网络权限） | +5 |
| 不访问其他应用数据 | +5 |

### 来源可信度（20分）

| 检查项 | 分值 |
|--------|------|
| GitHub 作者账号年龄 > 1年 | +5 |
| 作者有其他活跃项目 | +3 |
| 仓库有 README 和 LICENSE | +4 |
| 仓库有持续提交记录（非一次性提交） | +4 |
| 仓库未被标记为恶意 | +4 |

### 元数据完整性（10分）

| 检查项 | 分值 |
|--------|------|
| 有清晰的 Skill 描述 | +3 |
| 有使用文档/示例 | +3 |
| 有开源许可证 | +2 |
| 有版本号管理 | +2 |

### 社区认可度（10分）

| 检查项 | 分值 |
|--------|------|
| GitHub Star >= 100 | +4 |
| GitHub Star >= 10 | +2 |
| 有社区贡献者（非单人项目） | +3 |
| 近30天有更新 | +3 |

### 算分公式

```
总分 = 代码安全性得分 + 权限合规性得分 + 来源可信度得分 + 元数据完整性得分 + 社区认可度得分

安全等级：
  S级（95-100）：极致安全 ⭐⭐⭐⭐⭐
  A级（85-94）：非常安全 ⭐⭐⭐⭐
  B级（70-84）：基本安全 ⭐⭐⭐
  C级（50-69）：需谨慎使用 ⭐⭐
  D级（<50）：存在风险 ⭐（或不予上架）
```

### 评分展示 UI 文案

```
┌─────────────────────────────────────┐
│  🛡️ 安全评分 95/100               │
│  ⭐⭐⭐⭐⭐ 极致安全                │
│                                     │
│  ✅ 通过 12 项安全检测              │
│  ✅ 无危险命令                      │
│  ✅ 无提权行为                      │
│  ✅ 开源可审计                      │
│                                     │
│  📋 查看完整安全报告 →              │
└─────────────────────────────────────┘
```

**推荐文案变体：**
- "经过 12 重安全检测，安全评分 95 分"
- "已通过代码审查 · 权限合规 · 来源验证"
- "技能宝安全认证 — 放心使用"

### 实现难度与时间

- **实现难度**：低
- **所需时间**：3-5 天
- **技术要求**：正则匹配 + GitHub API + 简单评分逻辑
- **推荐优先级**：⭐⭐⭐⭐⭐（P0，必须第一个做）

## 方案二：借力评分（中等成本）

### 在方案一基础上增加的维度

| 新增维度 | 权重调整 | 数据来源 |
|----------|----------|----------|
| **第三方安全评估** | 15% | OpenSSF Scorecard、Snyk |
| **社区安全反馈** | 10% | 用户举报 + 评论 |

### 调整后的权重分配

| 维度 | 权重 |
|------|------|
| 代码安全性（静态分析） | 25% |
| 权限合规性 | 20% |
| 来源可信度 | 15% |
| **第三方安全评估（新增）** | **15%** |
| 元数据完整性 | 5% |
| 社区认可度 | 10% |
| **社区安全反馈（新增）** | **10%** |

### 第三方安全评估（15分）

**OpenSSF Scorecard 接入：**
- 通过 OpenSSF Scorecard API 获取项目评分
- 检查维度包括：是否启用分支保护、是否有 CI/CD、依赖是否锁定、是否有安全策略文件等
- OpenSSF 评分 >= 7/10 → +10分
- OpenSSF 评分 5-7/10 → +6分
- OpenSSF 评分 < 5/10 或无评分 → +2分

**Snyk 漏洞检查：**
- 检查 Skill 依赖链中是否有已知漏洞（CVE）
- 零已知漏洞 → +5分
- 低危漏洞 → +3分
- 中/高危漏洞 → +0分

### 社区安全反馈（10分）

**用户举报机制：**
```
举报类型：
├── 恶意行为（破坏数据、窃取信息）    → 触发紧急审查
├── 权限滥用（请求不必要的权限）      → 扣2分/次，累计
├── 误导性描述（功能与描述不符）      → 扣1分/次，累计
└── 其他安全顾虑                      → 人工审核
```

**评分规则：**
- 无任何举报记录 → +10分
- 有举报但经审核为误报 → +8分
- 有有效举报但已修复 → +5分
- 有未处理的有效举报 → +0分（并下架审查）

**反垃圾机制：**
- 同一用户对同一 Skill 仅能举报一次
- 恶意举报者（多次误报）降低举报权重
- 举报需提供具体描述，不接受空白举报

### 算分公式

```
总分 = Σ(各维度得分 × 权重) × 100

调节因子：
- 若有紧急举报未处理：总分 × 0.5
- 若被 OpenSSF 标记高风险：总分 × 0.7
- 若有已知 CVE 未修复：总分 × 0.8
```

### 实现难度与时间

- **实现难度**：中等
- **所需时间**：2-3 周
- **技术要求**：方案一 + OpenSSF API 对接 + Snyk API 对接 + 举报系统后端
- **成本**：Snyk 免费版可覆盖基本需求；OpenSSF Scorecard 完全免费
- **推荐优先级**：⭐⭐⭐⭐（P1，方案一上线后第二阶段迭代）

## 方案三：深度评分（高成本）

### 在方案二基础上增加的维度

| 新增维度 | 权重调整 | 数据来源 |
|----------|----------|----------|
| **AI 代码审查** | 15% | Claude/GPT 深度分析 |
| **沙箱运行测试** | 10% | Docker 隔离环境 |
| **定期安全扫描** | 5% | 周期性自动复查 |

### 调整后的权重分配

| 维度 | 权重 |
|------|------|
| 代码安全性（静态分析） | 15% |
| 权限合规性 | 10% |
| 来源可信度 | 10% |
| 第三方安全评估 | 10% |
| **AI 代码审查（新增）** | **15%** |
| **沙箱运行测试（新增）** | **10%** |
| 元数据完整性 | 5% |
| 社区认可度 | 10% |
| 社区安全反馈 | 10% |
| **定期安全扫描（新增）** | **5%** |

### AI 代码审查（15分）

**审查流程：**
1. 将 Skill 完整代码提交给 AI（Claude API）
2. AI 从以下角度分析：
   - **意图分析**：代码实际行为是否与描述一致
   - **隐蔽风险**：是否有经过混淆的恶意代码
   - **数据流分析**：用户数据是否被不当传输
   - **逻辑漏洞**：是否存在可被利用的逻辑缺陷
3. AI 输出结构化审查报告

**评分规则：**
- AI 审查通过，无风险点 → +15分
- AI 发现低风险问题 → +10分
- AI 发现中风险问题 → +5分
- AI 发现高风险问题 → +0分（触发人工复审）

**Prompt 模板（供 AI 审查使用）：**
```
你是一个代码安全审查专家。请分析以下 Claude Code Skill 的代码，从以下维度评估：
1. 代码实际行为是否与描述一致
2. 是否存在隐蔽的恶意行为（数据窃取、远程代码执行、文件破坏）
3. 是否有不必要的权限请求
4. 是否有混淆或编码后的可疑代码
5. 数据流向是否安全（用户数据是否被发送到外部）

请输出 JSON 格式的审查结果：
{
  "risk_level": "none|low|medium|high|critical",
  "issues": [...],
  "recommendation": "pass|review|reject",
  "score": 0-15
}
```

### 沙箱运行测试（10分）

**测试环境：**
- Docker 隔离容器，限制网络、文件系统、CPU/内存
- 模拟用户使用场景，运行 Skill 的核心功能
- 监控 Skill 运行时行为

**监控指标：**
| 监控项 | 正常行为 | 可疑行为 |
|--------|----------|----------|
| 网络请求 | 仅访问声明的域名 | 访问未声明的外部地址 |
| 文件操作 | 仅在工作目录内 | 访问系统文件或用户敏感目录 |
| 进程行为 | 无子进程或仅声明的进程 | 启动未声明的后台进程 |
| 资源消耗 | CPU/内存在合理范围 | 异常高资源占用（挖矿特征） |

**评分规则：**
- 沙箱运行无异常 → +10分
- 有轻微偏差但可解释 → +6分
- 有可疑行为 → +0分（触发人工审查）
- 沙箱运行崩溃/超时 → +3分（标记为"未完成测试"）

### 定期安全扫描（5分）

**扫描策略：**
- 精选库 Skill：每周自动重新评分
- 普通 Skill：每月自动重新评分
- 当 Skill 代码更新时：触发即时重新评分

**评分规则：**
- 持续 3 个月无安全问题 → +5分（"长期安全"徽章）
- 持续 1 个月无安全问题 → +3分
- 最近一次扫描发现新问题 → +0分

### 算分公式

```
基础分 = Σ(各维度得分 × 权重) × 100

时间衰减因子 = max(0.7, 1 - (距上次扫描天数 / 180))

最终得分 = 基础分 × 时间衰减因子

额外标记：
- 🏆 金牌认证：连续3个月 S级 + AI审查通过 + 沙箱测试通过
- 🛡️ 安全认证：当前评分 >= 85 + 无未处理举报
- ⚠️ 待审查：有新举报或评分下降超过10分
- 🚫 已下架：评分 < 50 或有高危问题
```

### 实现难度与时间

- **实现难度**：高
- **所需时间**：1-2 个月
- **技术要求**：方案二 + AI API 对接 + Docker 沙箱环境 + 定时任务系统 + 监控告警
- **成本**：AI API 调用费用（每个 Skill 审查约 $0.05-0.15）+ 沙箱服务器费用
- **推荐优先级**：⭐⭐⭐（P2，产品成熟后的终极形态）

## 三套方案对比总览

| 对比维度 | 方案一：基础评分 | 方案二：借力评分 | 方案三：深度评分 |
|----------|-----------------|-----------------|-----------------|
| **实现难度** | 低 | 中等 | 高 |
| **开发时间** | 3-5 天 | 2-3 周 | 1-2 个月 |
| **运营成本** | 几乎为零 | 低（API 免费额度） | 中等（AI + 服务器） |
| **安全覆盖率** | 60% | 80% | 95%+ |
| **营销价值** | "通过 12 重安全检测" | "接入国际安全数据库" | "AI + 沙箱双重防护" |
| **用户感知** | 有基本信任感 | 专业可靠感 | 极致安全感 |
| **推荐优先级** | P0 立即做 | P1 第二阶段 | P2 长期规划 |

## 推荐执行路径

```
第1周：上线方案一（基础评分）
  → 快速建立"安全可感知"的品牌认知
  → 所有 Skill 都有评分，精选库 Skill 评分高

第2-4周：迭代到方案二（借力评分）
  → 接入 OpenSSF + Snyk，评分更有公信力
  → 上线举报机制，建立社区信任

第2-3个月：逐步引入方案三（深度评分）
  → 先对精选库 3000-5000 个 Skill 做 AI 审查
  → 逐步扩展沙箱测试覆盖范围
  → 推出"金牌认证"标识，作为高端营销素材
```

### 对用户
- "技能宝每个 Skill 都经过 N 重安全检测，安全评分透明可见"
- "看到安全评分再安装，用 AI 技能也能放心"
- "不安全的 Skill 我们不上架"

### 对开发者
- "提交到技能宝的 Skill 自动获得安全认证，提升用户信任度"
- "安全评分高的 Skill 获得更多推荐流量"

### 对媒体/PR
- "技能宝是首个为 AI Skill 提供安全评分的平台"
- "我们接入了 OpenSSF 国际安全标准，为每个 Skill 建立安全档案"
